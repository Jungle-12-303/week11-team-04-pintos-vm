## 커밋 컨벤션

<aside>
✅

### 1. 작업 유형 지정

- 작업 유형은 소문자로만 구성.
    
    
    | 작업 유형 | 의미 |
    | --- | --- |
    | `feat` | 새로운 기능 추가 |
    | `fix` | 버그 수정 |
    | `docs` | 문서 수정 |
    | `style` | 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우 |
    | `refactor` | 코드 리팩토링 |
    | `test` | 테스트 코드, 리팩토링 테스트 코드 추가 |
    | `chore` | 의존성 수정, 간단한 변경 |
    | `db` | 스키마 변경 사항 |

### 2. 가능한 한 하나의 작업 단위로 커밋

### 3. 메세지는 한글로 작성

```
# 콜론 + 공백 1칸
feat: 구글 로그인 구현
```

### 4. 변경 사항이 많을 경우 추상화해서 작성하고 설명 추가

```
refactor: 피드 조회 조건 추가

deletedAt이 null인 조건
cursor보다 이전 조건
...
```

</aside>

## 브랜치 네이밍 컨벤션

<aside>
✅

### 1. 작업 유형 지정

- 작업 유형은 소문자로만 구성.
    
    
    | 작업 유형 | 의미 |
    | --- | --- |
    | `feat` | 새로운 기능 추가 |
    | `fix` | 버그 수정 |
    | `docs` | 문서 수정 |
    | `style` | 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우 |
    | `refactor` | 코드 리팩토링 |
    | `test` | 테스트 코드, 리팩토링 테스트 코드 추가 |
    | `chore` | 의존성 수정, 간단한 변경 |
    | `db` | 스키마 변경 사항 |

### 2. 브랜치 이름은 - 작업 유형/작업 내용 (영문)

```
feat/implement-google-login
```

</aside>

## 브랜치 전략

- main: 릴리즈 브랜치
- dev: 실제 작업 브랜치 모든 작업은 dev 브랜치에서 분기
- hotfix: main 브랜치에서 오류가 발생시 분기해 작업용 브랜치
